# PRD 07 — deployed-tfvars snapshot per workspace phase

> Captures the exact var-file inputs that produced the current state, so a workspace can be re-created, audited, or handed off without losing the "what did I actually deploy" record. Estimated effort: small (~150 LOC + tests).

## Why

`roksbnkctl` today layers tfvars from multiple sources at apply time:

- `config.yaml`-derived vars (from `internal/config/workspace.go`)
- the workspace-local `terraform.tfvars.user` if present
- the cluster-phase override (`state-cluster/cluster-phase-override.tfvars`, generated per-apply at `internal/cli/cluster_phase.go:239`)

Terraform combines these at apply time and the resulting state reflects the union. But after the apply, **only the state file persists** — the var-file chain is reconstructable only by re-running `roksbnkctl` against the same workspace config. If the user:

- loses or regenerates `config.yaml` (manual edit, machine migration)
- wants to re-create the workspace on a teammate's host
- wants to audit "what variables did I deploy 90 days ago?"
- has a corrupted state and needs to `terraform import` against the same inputs

…they have no on-disk record of the effective inputs. They reconstruct from memory, from git history of `config.yaml` (if the workspace dir is committed), or from `terraform state show` (which only shows derived values, not the inputs that produced them).

The fix is small: at the end of every successful `terraform apply` for a phase, write a single canonical tfvars file capturing the effective inputs to that apply. Workspace-local, phase-scoped, overwritten each apply.

## Goal

After every successful cluster-phase or trial-phase `terraform apply`, write a file to the workspace's per-phase state directory containing the effective var-file contents used in that apply, in canonical HCL ordering.

| Phase | Trigger | File path |
|---|---|---|
| Cluster phase (`roksbnkctl cluster up`, `roksbnkctl up` for ShapeLegacySingle) | After `tf.Apply` returns nil | `~/.roksbnkctl/<workspace>/state-cluster/terraform.applied.tfvars` (cluster phase) — for ShapeLegacySingle, `~/.roksbnkctl/<workspace>/state/terraform.applied.tfvars` since both phases share the state dir |
| Trial phase (`roksbnkctl bnk up`) | After `tf.Apply` returns nil | `~/.roksbnkctl/<workspace>/state/terraform.applied.tfvars` |

The file is **idempotent** (same inputs → byte-identical output), **mode `0600`** (may contain semi-sensitive values), and **never read by `roksbnkctl` itself** — it's an output for the user, not an input the tool depends on.

## Design

### File contents

A single tfvars-format HCL file. One assignment per line. Variables sorted alphabetically. Each section preceded by a comment line documenting the source:

```hcl
# Generated by roksbnkctl <version> at <RFC3339 timestamp> after
# terraform apply on phase=<cluster|trial>. Re-generated each apply.
# Do not edit by hand — your changes will be overwritten.

# === from config.yaml ===
cluster_name = "canada-roks"
region       = "ca-tor"
resource_group_name = "default"
...

# === from terraform.tfvars.user (if present) ===
worker_count = 4
...

# === from cluster-phase override (cluster phase only) ===
deploy_bnk = false
```

Source-attribution comments matter because the same variable can appear in multiple sources; the "winner" (the value terraform actually used) is the last source in the chain. Source comments let the user trace why a particular value ended up here.

### Trigger point

`internal/exec/tf.Workspace.Apply` (the wrapper that runs `terraform apply -var-file=... -var-file=...`) returns nil on success. The caller — `runUp` for the v1.0.x flat path, `runClusterUp` for cluster phase, `runBnkUp` for trial phase — currently does its post-apply work (cluster-outputs.json refresh, etc.) and returns. The new snapshot write hooks in **after `Apply` returns nil and before the caller's own post-apply work**, so:

- a failed apply produces no snapshot (the existing file from the prior successful apply stays — represents the actual current state)
- a successful apply always produces a snapshot before any post-apply step runs
- the caller can read the snapshot if it needs to (e.g., cluster-outputs.json refresh wants the effective values)

A new helper `internal/config.WriteAppliedTFVars(workspace, phase, sources)` centralizes the write so all callers share one path.

### Sensitive value handling

Exactly one variable is redacted: `ibmcloud_api_key`. It's written as `<redacted>` with a comment:

```hcl
ibmcloud_api_key = "<redacted>"  # source: cred resolver, not persisted
```

This is the only variable today whose value comes from the cred resolver rather than being authored by the user in `config.yaml` or a tfvars file — so it's the only value the snapshot would expose that the user didn't explicitly put there. Every other variable is something the user already chose to write into their workspace config, and round-tripping it through the snapshot doesn't expose anything new.

The file is **mode `0600`** regardless. The contents may include workspace identifiers, region/account hints, and tunable values that aren't secret-secret but aren't world-readable-grade either. Tight permissions are cheap; defaults bias toward the safer side.

If a future tfvars surface introduces a new sensitive variable, the redaction list extends in code (one-line change) — no config knob needed. See open question 5 below; ruled out for v1.x.

### Source ordering

The order of source sections in the file is the same as terraform's `-var-file` evaluation order:

1. `config.yaml`-derived vars (the workspace's primary source of truth)
2. `terraform.tfvars.user` (user's per-workspace override, if present)
3. Phase-specific overrides (cluster-phase-override.tfvars on cluster phase)

This means the "last write wins" rule is the same as terraform's: if a variable appears in two sections, the later section's value is the one terraform used (and the one that should appear in `terraform.applied.tfvars` as the effective value — though the earlier sections retain the prior values for traceability).

Implementation note: rather than re-parsing the source tfvars files at snapshot time, `tf.Workspace.Apply` can pass the same `[]string` of var-file paths it gave to `terraform apply` directly to `WriteAppliedTFVars`, which reads each file in order, applies redaction, and emits the source-attributed snapshot. No re-derivation.

### Phase awareness on ShapeLegacySingle

`ShapeLegacySingle` workspaces have one state dir (`state/`) and `roksbnkctl up` (the v1.0.x flat command) applies both cluster and trial together. The snapshot path on Legacy is `state/terraform.applied.tfvars` and contains the union of all sources — no separate cluster vs trial split, since the legacy shape doesn't separate them.

The `# Generated by roksbnkctl ... on phase=<...>` comment in the file header records `phase=legacy-single` on Legacy workspaces so the reader doesn't mistake it for either a cluster-only or trial-only snapshot.

### What's NOT captured

- **Terraform defaults that weren't overridden.** If `variable "foo" { default = "bar" }` and the user never set `foo`, the snapshot doesn't include `foo = "bar"`. The snapshot is "what var-files said," not "what terraform's effective resolved values were." Capturing terraform defaults would require running `terraform output` against the variables block — out of scope for this PRD. Users who want defaults can `terraform console` against the live state dir.

- **Computed/dynamic values.** Same reasoning — anything that terraform evaluated from expressions (resource references, locals, data sources) isn't a var-file input and doesn't appear.

- **Environment variables (`TF_VAR_*`).** Terraform also accepts `TF_VAR_<name>=<value>` from the environment as a tfvars-equivalent. `roksbnkctl` doesn't set these today (everything goes via `-var-file`), so the snapshot can omit them. If a future cycle starts setting `TF_VAR_*` env vars, this PRD's snapshot path will need a parallel capture step.

## Tradeoffs

**Why overwrite-on-apply instead of dated snapshots?** Two reasons:

1. The user already has version-control options for history (commit `~/.roksbnkctl/<workspace>/` to git, use `restic`, etc.). Dated snapshots inside the workspace dir create their own retention/cleanup question that's better left external.
2. The "current state" file is the most actionable — it represents the inputs that produced the *current* deployed cluster. A directory of dated snapshots without a clearly-marked "current" file is more confusing than just `terraform.applied.tfvars` always being the current state.

If a user wants history, they can `cp terraform.applied.tfvars terraform.applied.tfvars.$(date +%s)` before each apply or wire that into a pre-apply hook in a future cycle.

**Why mode 0600 when only `ibmcloud_api_key` is redacted?** The non-redacted contents include workspace identifiers, account region, resource group, cluster name, and various tunable values — not secrets in the credential sense, but not appropriate for `/etc/skel`-style world-readability either. Mode `0600` is the cheap safe default. If a user wants to share the file, they `chmod` and share; the tool doesn't make that decision for them.

**Why phase-scoped instead of workspace-level union?** ShapeSplit is the modern shape — cluster phase and trial phase are independent applies with independent state, and a union file would have to either pick one phase's view or annotate every variable with phase. Phase-scoped files match the shape and let the user grep by phase trivially. The Legacy case keeps the union since it's already a union at the state level.

## Anti-patterns to avoid

1. **Don't make `terraform.applied.tfvars` an *input* to subsequent applies.** It's an output. Adding it to the `-var-file` chain on the next apply would create a feedback loop where redacted values get written back as the literal string `<redacted>`. The chain stays: `config.yaml`-derived → `terraform.tfvars.user` → phase overrides. Period.

2. **Don't write the snapshot on a failed apply.** A snapshot must correspond to a state that actually got applied. Writing on partial-failure would leave the user with a snapshot that doesn't match anything terraform managed.

3. **Don't capture state-derived values.** Use the var-file inputs only. State-derived capture is a separate (much larger) PRD — call it "post-apply state introspection" if it ever becomes a thing.

4. **Don't fail the apply if the snapshot write fails.** The apply succeeded; the snapshot is a nice-to-have output. Log a `warn` to stderr (`could not write terraform.applied.tfvars: <err>`) and continue. The apply's exit code reflects the apply, not the snapshot's bookkeeping.

## Resolved design decisions

Locked in during PRD review:

1. **Trigger**: successful `terraform apply` only. Plan flows do not snapshot — the file name `terraform.applied.tfvars` would be misleading if a plan-time write existed.

2. **Destroy behavior**: `cluster down` / `bnk down` leave the prior `up`'s snapshot intact. The snapshot then represents "what was last deployed" rather than "what's deployed right now." State and snapshot diverge after destroy — that divergence is itself a useful signal (the file's mtime tells you when the last apply was; the absence of state tells you it's since been torn down). Re-apply with the same inputs is the natural workflow this enables.

3. **`.gitignore` handling**: `roksbnkctl` does not touch `.gitignore`. Chapter 6 (workspaces) will document that the file is safe to commit *after* the user verifies the redaction matches their threat model, with the standard reminder that the workspace dir may contain other semi-sensitive material.

4. **Redaction list**: hardcoded to exactly `ibmcloud_api_key`. No pattern matching, no config knob, no extension surface in v1.x. If a future tfvars surface adds a new credential-grade variable, the redaction list extends in code (one-line addition).

## Open questions

1. **Should `ops install` and `ops uninstall` also snapshot?** These don't run terraform, but they do change cluster-side state (Kubernetes objects, IAM trusted profile). A snapshot for them would have a different shape (no tfvars — instead a record of the SA annotations, the Secret state, the `--trusted-profile=…` value used). **Recommendation: out of scope** for this PRD — file as a follow-up PRD if there's user demand.

## Acceptance criteria (Sprint NN — TBD)

- `internal/config/applied_tfvars.go` (new file): `WriteAppliedTFVars(workspace, phase, sources []string) error`. Reads each source path, applies the redaction list, writes the source-attributed snapshot to the phase's state dir at mode `0600`. Returns nil on success, error (logged-not-failed by callers) on write failure.

- `internal/exec/tf.Workspace.Apply` calls `WriteAppliedTFVars` after a successful apply, passing the same `varFiles` slice it gave to `terraform apply`. Failure mode: log-and-continue (per anti-pattern 4).

- Unit tests against fixture workspaces with all three shape coverage (Empty/ClusterOnly/Split) plus Legacy, asserting:
  - File written at the expected path with mode `0600`
  - Contents include source-attribution comments
  - `ibmcloud_api_key` (and only that variable) emitted as `<redacted>`
  - Variable sort order is alphabetic within each section
  - Re-running apply with identical inputs produces a byte-identical file (idempotency)
  - Destroy flow does NOT modify the prior snapshot

- Chapter 6 (workspaces) updated with a §"`terraform.applied.tfvars` — what's deployed right now" subsection covering the file's location, what it captures, the redaction list, and the "safe to commit after review" guidance.
