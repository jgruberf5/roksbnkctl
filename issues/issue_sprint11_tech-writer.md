## Issue 2: dogfooding — disaster recovery from `terraform.applied.tfvars` alone

**Severity**: low (discoverability — chapter 6 covers it but the limit-of-the-snapshot reminder is one paragraph in a `### What it's not` subsection)
**Status**: resolved (chapter 6 §"What it's **not**" + PRD 07 §"What's NOT captured" already document the gap)

### Scenario

I lost `config.yaml`. Can I reconstruct a working workspace from `terraform.applied.tfvars`?

### Walkthrough

The snapshot contains every variable the user explicitly set across the three sources (`config.yaml` → `terraform.tfvars.user` → cluster-phase override) in canonical HCL with alphabetic sort. The user can:

1. `cp ~/.roksbnkctl/<ws>/state-cluster/terraform.applied.tfvars` to a new workspace's `terraform.tfvars.user` (after redaction-aware editing — the `<redacted>` placeholder for `ibmcloud_api_key` must be replaced with their key resolver output, or removed so the cred resolver supplies it at apply time).
2. Reconstruct a minimal `config.yaml` from the `# === from config.yaml ===` section's keys (workspace name, region, resource group, cluster name).
3. Re-run `roksbnkctl init -w <ws>` against the reconstructed config and apply.

### Stuck-point

PRD 07 §"What's NOT captured" rules out **Terraform defaults that weren't overridden**. If `variable "foo" { default = "bar" }` in the embedded TF module and the user never explicitly set `foo`, the snapshot omits `foo` entirely. A re-apply from the snapshot alone produces the same effective `foo = "bar"` *only if* the embedded TF module's default for that variable hasn't changed between the original apply and the re-apply. Across `roksbnkctl` versions, embedded TF defaults can drift; a user reconstructing from a snapshot taken under `v1.2.0` against a `v1.4.0` binary may get different defaults silently.

Chapter 6 §"What it's not" calls this out (line 85): *"Not a record of Terraform defaults. If `variable "foo" { default = "bar" }` and the user never set `foo`, the snapshot omits `foo` entirely."* The reader who reaches this paragraph understands the caveat. The stuck-point is purely about reaching the paragraph — `### What it's not` is a sub-section several scrolls down from `### Where it lives`, and a user in disaster-recovery mode is likely to start with the worked example and the path.

### Recommendation (carry to v1.4.x polish backlog — not v1.4.0 blocker)

Either:

- Add one sentence to the §"Worked example" prose (line 116) along the lines of *"Re-applying from this snapshot alone reconstructs the inputs the user wrote; embedded Terraform module defaults are not captured (see §'What it's not')."*
- Or, more thorough: a short §"Reconstructing a workspace from the snapshot" sub-section walking through the three steps above with the defaults caveat embedded.

Neither is required for v1.4.0 — the existing prose is complete. This is a discoverability nudge, not a correctness gap.

## Issue 4: dogfooding — team handoff with redacted API key

**Severity**: low (discoverability — the redacted inline comment doesn't cross-link to chapter 14)
**Status**: resolved (chapter 6 §"Redaction" already cross-links)

### Scenario

A teammate wants to re-create my workspace from `terraform.applied.tfvars`. Is the file enough?

### Walkthrough

Yes, modulo the API key:

1. The teammate copies the file, edits the `ibmcloud_api_key = "<redacted>"` line to point at their own key (or removes the line entirely so their cred resolver supplies it at apply time).
2. Every other variable round-trips verbatim — region, resource group, cluster name, worker count, etc.
3. The teammate creates a workspace with the matching name, populates `config.yaml` from the snapshot's `# === from config.yaml ===` section, and runs `roksbnkctl cluster up`.

### Stuck-point

The redacted line's inline comment reads `# source: cred resolver, not persisted` (`internal/config/applied_tfvars.go:205`). A teammate who hasn't read chapter 14 may not know what "cred resolver" means or where to learn how to supply their own key.

Chapter 6 §"Redaction" (line 74) cross-links to chapter 14: *"It's the only var whose value comes from the [cred resolver](./14-credentials-resolver.md) rather than being authored…"* — so a reader who reads chapter 6 before opening the file is well-served. The stuck-point only bites a reader who opens the file *without* reading chapter 6 first (e.g., a teammate handed the file out-of-band).

### Recommendation (carry to v1.4.x polish backlog — not v1.4.0 blocker)

The inline-comment text in the binary is one line of code (`internal/config/applied_tfvars.go:205`). A future cycle could extend it to point at a discoverable doc URL — e.g.:

```hcl
ibmcloud_api_key = "<redacted>"  # source: cred resolver, not persisted (see https://<docs>/14-credentials-resolver.html)
```

Out of scope for v1.4.0 — the URL would need to be a documented constant in the cred resolver's package + would need to survive `mdbook` URL rewriting. Filing as a Sprint 12+ polish surface for the architect to scope.
