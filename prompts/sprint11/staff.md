## Tasks (priority order)

### 1. `internal/config/applied_tfvars.go` (new file)

Signature: `func WriteAppliedTFVars(workspace, phase string, sources []string) error`

Where:
- `workspace` — workspace name (e.g. `canada-roks`).
- `phase` — one of `"cluster"`, `"trial"`, or `"legacy-single"`. Used for the header comment + to pick the target state dir.
- `sources` — ordered `[]string` of var-file paths terraform was given (same slice that came from `Workspace.varFiles`). Read each in order; later sections override earlier sections per terraform semantics.

Implementation outline:

1. Resolve target path per phase:
   - `"cluster"` → `<WorkspaceClusterStateDir>/terraform.applied.tfvars`
   - `"trial"` → `<WorkspaceStateDir>/terraform.applied.tfvars`
   - `"legacy-single"` → `<WorkspaceStateDir>/terraform.applied.tfvars` (same as trial path, since legacy collapses both into one state dir)

2. For each `sources[i]`:
   - Read the file (if missing, skip — best-effort, but log a warning to stderr so the user knows their tfvars source was missing).
   - Parse the variable assignments (you can use a minimal HCL parser — `github.com/hashicorp/hcl/v2/hclsimple` is already in module deps via go.sum, check first; or a regex-based scan if HCL parsing is overkill).
   - Collect into a `map[string]string` per source, preserving the source label for the section comment.

3. Build the output:
   - Header comment: `# Generated by roksbnkctl <version> at <RFC3339 timestamp> after terraform apply on phase=<phase>. Re-generated each apply. Do not edit by hand — your changes will be overwritten.`
   - For each source section, a section header comment (`# === from <source label> ===`) followed by alphabetically-sorted `<name> = <value>` lines.
   - For `name == "ibmcloud_api_key"`, write `ibmcloud_api_key = "<redacted>"  # source: cred resolver, not persisted` instead of the actual value.
   - Trailing newline.

4. Write the file with mode `0600` via `os.WriteFile`. Atomic-rename pattern if you want belt-and-suspenders (write to tempfile in same dir, rename) — optional but cheap.

5. Return `nil` on success, error on filesystem failure. Callers will log-and-continue (don't fail apply).

The "source label" for the section comment should be human-friendly:
- `config.yaml`-derived path (look for `roksbnkctl-config.tfvars` or whatever the generated filename is in the state dir) → `from config.yaml`
- `terraform.tfvars.user` → `from terraform.tfvars.user`
- `cluster-phase-override.tfvars` → `from cluster-phase override`
- anything else → use the basename

You'll need to map source paths to labels. The mapping is small enough to hardcode.

### 2. Apply hook in `internal/tf/terraform.go`

In `Workspace.Apply(ctx, extraVarFiles...)`, after the existing terraform-apply call returns nil:

```go
phase := workspacePhase(w)  // small helper — see below
sources := w.varFiles(extraVarFiles...)
if err := config.WriteAppliedTFVars(w.name, phase, sources); err != nil {
    fmt.Fprintf(w.stderr, "warning: could not write terraform.applied.tfvars: %v\n", err)
}
return nil
```

`workspacePhase` is a small helper that maps the `Workspace`'s state dir to a phase string. The cluster-phase state dir contains `state-cluster` in its path; trial-phase contains `state` (without the `-cluster` suffix). ShapeLegacySingle uses the trial state dir as well; you can distinguish via the workspace's `DetectShape` result or by checking whether the cluster-phase-override is present in `sources`.

Edge case: if `Apply` is called for a phase the helper can't classify, default to `"trial"` and let the file land at the trial-phase path. (This shouldn't happen on the documented call paths but the fallback prevents a hard error.)

### 3. Unit tests

File: `internal/config/applied_tfvars_test.go` (new).

Test cases (table-driven if cleaner):

- **Shape coverage** — Empty / ClusterOnly / Split / Legacy. Each shape exercises the snapshot path for the relevant phase(s); for Empty there's no apply so no snapshot is expected (test that the path stays empty).
- **Source attribution** — given three source files, the output has three `# === from <label> ===` section headers in the correct order.
- **Alphabetic sort within section** — feed variables in `c=…,a=…,b=…` order to a single source, assert output is `a`, `b`, `c`.
- **Redaction** — `ibmcloud_api_key = "actual-secret"` in a source becomes `ibmcloud_api_key = "<redacted>"` in the output, with the explanatory comment. ALL OTHER variable names pass through verbatim (negative test: a hypothetical `database_password = "x"` does NOT get redacted — only the literal name `ibmcloud_api_key` triggers).
- **Idempotency** — same inputs produce a byte-identical file on re-run.
- **File permissions** — `os.Stat` confirms mode `0600`.
- **Missing source file** — `WriteAppliedTFVars` doesn't crash; the section either gets skipped or marked `# === from <label> (missing) ===`. Test the chosen behavior.
- **Destroy doesn't mutate** — separate test exercising the `Destroy` path; assert the snapshot from a prior `Apply` is unchanged after `Destroy` runs. (This is a behavioral check, not a `WriteAppliedTFVars` direct call — exercises the absence of a hook in `Destroy`.)

Use the existing fixture-copy pattern from `internal/cli/bnk_phase_test.go` — `t.TempDir()`, `ROKSBNKCTL_HOME` env override, minimal `SaveWorkspace`, then `WriteAppliedTFVars` direct call.
